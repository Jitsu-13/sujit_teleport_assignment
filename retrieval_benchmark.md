# Retrieval Benchmark: Strategy A vs Strategy B

Generated by running `python -m src.pipeline` against the 10-document technical corpus.

---

## How the Two Strategies Work

| | Strategy A — Raw Vector Search | Strategy B — AI-Enhanced Retrieval |
|---|---|---|
| **Query processing** | Embed the query as-is | `MockGenerativeModel` rewrites the query first, then embed |
| **Expansion** | None | Keyword-triggered domain expansion (e.g. "peak load" → adds *auto-scaling, horizontal scaling, burst handling, …*) |
| **Embedding** | `LocalEmbeddingModel` (sentence-transformers `all-MiniLM-L6-v2`) | Same, applied to the expanded query |
| **Search** | `VectorStore.search()` — FAISS `IndexFlatIP`, cosine similarity | Same index and metric |

## Design Decisions

### Similarity Metric: Cosine vs Euclidean

**We use cosine similarity (implemented as inner product on L2-normalised vectors).**

| | Cosine Similarity | Euclidean Distance |
|---|---|---|
| **Magnitude sensitivity** | None — only direction matters | Yes — longer vectors score higher |
| **Short vs long passages** | Treated equally | Short chunks penalised |
| **Unit-vector equivalence** | `cos(a,b) = a·b` (fast FAISS IP kernel) | Requires separate normalisation step |
| **Vertex AI default** | Dot-product (= cosine for unit vectors) | Not the default |
| **FAISS index** | `IndexFlatIP` | `IndexFlatL2` |

Cosine similarity is the right choice for document retrieval because passage length varies widely — a two-sentence definition and a six-sentence explanation of the same concept should score equally if their semantic direction matches. Euclidean distance would penalise the shorter passage simply because its vector has smaller magnitude.

All embeddings are L2-normalised at production time inside `LocalEmbeddingModel.embed()`, so `a·b = cos(a,b)` without any extra computation. This also means no normalisation step is needed at query time.

### Migrating to Vertex AI Vector Search (Matching Engine) in Production

The local FAISS implementation maps directly to Vertex AI Vector Search with minimal code changes:

**1. Replace `LocalEmbeddingModel` with the managed gecko model**
```python
# Local (current)
from src.embeddings import LocalEmbeddingModel
embedder = LocalEmbeddingModel()
vectors = embedder.embed(texts)

# Production
from vertexai.language_models import TextEmbeddingModel
model = TextEmbeddingModel.from_pretrained("textembedding-gecko@003")
embeddings = model.get_embeddings(texts)
vectors = [e.values for e in embeddings]
```

**2. Replace `VectorStore` with a Matching Engine index**
```python
import google.cloud.aiplatform as aip

# Create index (one-time)
index = aip.MatchingEngineIndex.create_tree_ah_index(
    display_name="rag-index",
    dimensions=768,            # gecko output dimension
    approximate_neighbors_count=10,
    distance_measure_type="DOT_PRODUCT_DISTANCE",  # matches cosine on unit vectors
)

# Deploy to endpoint
endpoint = aip.MatchingEngineIndexEndpoint.create(display_name="rag-endpoint")
endpoint.deploy_index(index=index, deployed_index_id="rag_deployed")

# Query
neighbours = endpoint.find_neighbors(
    deployed_index_id="rag_deployed",
    queries=[query_vector],
    num_neighbors=3,
)
```

**3. Replace `MockGenerativeModel` with Gemini**
```python
# Local (current)
from src.mocks import MockGenerativeModel
model = MockGenerativeModel("gemini-1.5-pro")

# Production
import vertexai
from vertexai.generative_models import GenerativeModel
vertexai.init(project="your-project", location="us-central1")
model = GenerativeModel("gemini-1.5-pro")
```

**Migration checklist:**

| Component | Local | Production |
|-----------|-------|------------|
| Embeddings | `sentence-transformers all-MiniLM-L6-v2` (384-d) | `textembedding-gecko@003` (768-d) |
| Vector index | `faiss.IndexFlatIP` | Vertex AI Matching Engine (ScaNN, ANN) |
| Similarity metric | Cosine via inner product | `DOT_PRODUCT_DISTANCE` (same) |
| Query expansion | `MockGenerativeModel` | `GenerativeModel("gemini-1.5-pro")` |
| Scaling | Single process | Fully managed, auto-scales to billions of vectors |
| Latency at scale | O(N) brute force | Sub-10ms approximate nearest neighbour |
| Cost | Free (local) | Per-query and per-node pricing |

The `VectorStore` and `StrategyA`/`StrategyB` interfaces are already abstract enough that only the constructor arguments change — the retrieval logic, prompt templates, and benchmark runner are production-ready as-is.
